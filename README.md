Reflection
How can I ensure that my code, program, or software is functional and secure?
Functional correctness starts with writing tests that define expected behavior before assuming the implementation is right. In this course, I built JUnit 5 test suites for the Contact, Task, and Appointment services that tested both the happy path and boundary conditions — null inputs, oversized strings, duplicate IDs, and invalid states. That discipline of testing edge cases, not just normal flow, is what surfaces the bugs that actually matter. Security at the code level follows a similar principle: constrain inputs at the boundary, reject anything that doesn't meet spec, and never let bad data propagate into the system. The field validation logic in ContactService — enforcing non-null, length-limited fields and rejecting duplicate contact IDs — reflects that thinking directly.
How do I interpret user needs and incorporate them into a program?
In CS 320, user needs arrived as written requirements: specific field constraints, allowed operations, and behavioral rules for each service. My job was to translate those into both implementation logic and test cases that verified the implementation honored them. I treated the requirements as a contract — if a contact ID must be non-null and no longer than 10 characters, that constraint appears in the constructor, in the service layer, and in at least two test cases. That traceability between requirement and test is how you confirm the software actually does what was asked, not just what seemed reasonable at the time.
How do I approach designing software?
I start with the constraints and work inward. For these projects, that meant understanding what the data model had to enforce, what the service layer had to guarantee, and what the tests had to verify — in that order. Keeping each class focused on a single responsibility made the design easier to test and easier to reason about. ContactService doesn't know anything about Tasks; each service owns its own state and validation. That separation isn't just good practice for maintainability — it's what makes unit testing tractable in the first place. If everything is coupled, you can't test anything in isolation.
